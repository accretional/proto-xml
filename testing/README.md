# testing/

Test harness for `proto-xml`. Three suites, all driven from the fixtures
under `data/`.

```
testing/
  validation/    one parametrised correctness test across every fixture
  fuzz/          go native fuzz targets seeded from data/ + edge cases
  benchmarks/    Decode/Encode/RoundTrip benchmarks across every fixture
```

Run from the repo root via the standard scripts (never invoke `go test` ad-hoc
for CI — see `AGENTS.md`):

- `./test.sh` — runs unit tests, validation, a fuzz smoke pass per target
  (`FUZZ_TIME` env, default `3s`), and `-bench=. -benchtime=1x` benchmarks.
- `./LET_IT_RIP.sh` — wraps `test.sh`. Required before commit/push.

## Strategy

Every fixture under `data/` is exercised against the same contract:

1. `Decode(src)` succeeds and returns a non-nil `DocumentElement`.
2. `XmlDocumentWithMetadata.RawBytes` equals `src` byte-for-byte.
3. `EncodeMetadata(md, {UseRawBytes: true})` reproduces `src` exactly.
4. `Encode(md.Document)` (structural re-emit) is parseable by `Decode` again.
5. The XML version round-trips on the structural path.

Fuzzing inverts the contract: any input — including malformed bytes — must
not panic `Decode`. For inputs that `Decode` accepts as well-formed,
structural re-encode + re-decode must also not panic and must succeed.

Benchmarks measure throughput (`SetBytes(len(file))`) and allocations for
`Decode`, structural `Encode`, raw-bytes `Encode`, and full round-trip.

## Known discrepancies and limitations

These are deliberate scope limits. Each one is referenced from the codec
source so it stays grepable.

- **XML 1.1 parsing.** `encoding/xml` only accepts `version="1.0"`. We
  detect the original declaration, store `XmlVersion = 1_1` on the proto,
  then patch the in-memory bytes to `1.0` for parsing only
  (`patchXMLVersion` in `internal/xmlcodec/decode.go`). RawBytes still
  preserves the original. Effect: XML 1.1-only name characters and the C1
  control-character requirements are not validated by the parser.
- **Non-UTF-8 character encodings.** `CharsetReader` is a no-op pass-through.
  Non-UTF-8 byte streams will fail to decode if they contain non-ASCII
  bytes outside the UTF-8 representation. UTF-16 BOMs are detected
  (`detectEncoding`) but not transcoded.
- **DOCTYPE internal subset.** `parseDoctypeDirective` captures only the
  root name, public/system IDs, and a `HasInternalSubset` flag. Internal
  subsets (`<!ELEMENT ...>`, `<!ATTLIST ...>`, `<!ENTITY ...>`) are not
  parsed and therefore not re-emitted on the structural encode path. Use
  the raw-bytes encode path if you need DOCTYPE preservation.
- **Entity resolution.** `encoding/xml` resolves the five predefined
  entities (`&lt; &gt; &amp; &quot; &apos;`) and numeric character refs
  during decode. `XmlText.RawPieces` is populated by re-scanning the
  source span so the original `&amp;` vs `&#38;` distinction survives the
  structural encode path. Custom entities defined in an internal subset
  resolve to the empty string (stdlib behaviour).
- **Self-closing detection.** `<foo/>` vs `<foo></foo>` is approximated:
  if the element has no children and the source contains a
  `<localName .../>` pattern, we mark it self-closing (`isSelfClosingTag`).
  Pathological inputs (e.g. comment text containing the same tag literal)
  can be misclassified — this is a metadata-only field; structural correctness
  is unaffected.
- **Per-attribute prefix capture.** Attribute prefixes are reconstructed
  from in-scope namespaces during structural encode. If the source uses a
  rare prefix shadowing pattern, the structural re-emit may pick a
  different prefix. RawBytes round-trip is unaffected.
- **Whitespace between prolog/epilog items** is not stored as a node; only
  RawBytes preserves it.
- **XInclude (`<xi:include/>`)** is not processed. The inclusion element
  is preserved as a regular element; no resolution is performed.

## Adding a new fixture

Drop the file in `data/handwritten/` (any `.xml` extension is picked up).
Programmatic fixtures live in `data/programmatic/` and are regenerated by
`setup.sh` from `cmd/gen-fixtures` whenever the directory is empty.

## Adding a new test

- Cross-fixture correctness assertion → extend `TestValidate` in
  `testing/validation/validation_test.go`.
- New panic invariant → add a `Fuzz*` function to `testing/fuzz/fuzz_test.go`.
- New benchmark dimension → add a `Benchmark*` to
  `testing/benchmarks/benchmarks_test.go`.
